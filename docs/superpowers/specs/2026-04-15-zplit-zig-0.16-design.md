# zplit Zig 0.16.0 Migration — Design Spec

**Date:** 2026-04-15
**Target version:** zplit v0.2.0 (Zig 0.16.0 compatibility release)
**Current state:** zplit 0.1.1 / Zig 0.15.x, src 約8252行 / 17ファイル
**Strategy:** ドキュメント駆動で仕様+Phase計画を先に確定。0.16.0 がローカルに降臨後、`zig build` のエラーを起点にPhase順で実行。

## 1. 目的と非目的

### 目的
- `minimum_zig_version` を `0.16.0` に上げる
- 0.16.0 の breaking changes（`std.Io` 移行、`std.fs` リネーム、`@intFromFloat` 廃止予定、`bufPrintZ` リネーム、グローバル環境/引数廃止 等）に追従
- 既存機能の挙動を1ビットも変えない（pure migration）

### 非目的
- 新機能追加・リファクタ・パフォーマンス改善
- `posix.*` 直叩きを `std.Io` 化すること（システム要求でlow-level必須なため低レベル維持）
- テスト追加（既存テストが通れば良し）

## 3. 設計判断

### 3.1 `Io` 引数の伝播戦略
zplit のエントリ `main` を 0.16の "Juicy Main" に切り替え:

```zig
pub fn main(init: std.process.Init) !void {
    const io = init.io;
    const gpa = init.gpa;
    // ...
}
```

そこから `Io` を必要箇所まで関数引数で渡す。グローバル化はしない。

伝播必要パス：
- `main → runServer → Server.init → ClientState/Tab/Pane` （Server がIO持つ）
- `main → runClient → Client.run` （Client がIO持つ）
- `session.save/load → file ops` （Session API に `io` 追加）
- `Config.load → file ops` （Config.load に `io` 追加）

`Server` / `Client` 構造体に `io: std.Io` フィールドを追加して、内部メソッドからは `self.io` で参照する形にすれば伝播コストを最小化できる。

### 3.2 posix 直叩きの維持
`server.zig` の epoll/socket/accept、`main.zig` の fork/execveZ、`pty.zig` の openpty 周りは 0.16 でも `std.posix` で同等APIを保持する見込み（zt メモ参照）。差分発生時は個別対応。

### 3.3 stdout/stderr の取り扱い
`client.zig:160` の `std.fs.File.stdout()` は client が tty として直接書き込む用途。0.16 では `init.io` 経由で stdout/stderr Writer を取る形になる見込み（要実機確認）。実装方針は実機で確定。

### 3.4 `bufPrintZ` 一括置換
`pty.zig` の環境変数構築9箇所は機械的に `std.fmt.bufPrintZ` → `std.fmt.bufPrintSentinel(buf, 0, fmt, args)` に置換可能。

### 3.5 `@intFromFloat` 置換
`pane.zig` の2箇所は `@round(f32値)` の結果を `u16` にしている。0.16 では:

```zig
// OLD
const v: u16 = @intFromFloat(@round(x));
// NEW
const v: u16 = @intCast(@as(i32, @trunc(@round(x)))); // 要確認、簡潔形があるかも
```

実機で最終形を確定。

## 4. データフロー / アーキテクチャ変更

なし。配線（誰がIOを持ち誰に渡すか）の追加だけ。

## 5. エラーハンドリング

0.16のリネームに追従:
- `error.RenameAcrossMountPoints` → `error.CrossDevice`
- `error.SharingViolation` → `error.FileBusy`
- `error.EnvironmentVariableNotFound` → `error.EnvironmentVariableMissing`

zplit 内で明示捕捉している箇所はないが、`switch (err)` 節で参照していたら追従する。grepで0件確認済み。

## 6. テスト戦略

- `zig build test` がパスすればOK（既存テスト維持）
- 手動回帰: `zplit new` / `zplit attach` / split / kill / detach / reattach の golden path
- セッション保存→再起動→復元の動作確認

## 7. Phase 分割（実装計画の骨格）

詳細は別途 `docs/superpowers/plans/2026-04-15-zplit-zig-0.16-migration.md` に記載予定。

| Phase | 内容 | 想定LOC変更 |
|---|---|---|
| P1 | `build.zig.zon` minimum_zig_version=0.16.0、`zig build` してエラー収集 | 1行 |
| P2 | `main.zig` を `Init` 受け取り化、`gpa/io/args/env` を `init` 経由に | ~30行 |
| P3 | `std.fmt.bufPrintZ` → `bufPrintSentinel` 一括置換 (pty.zig 9箇所) | ~10行 |
| P4 | `@intFromFloat` 置換 (pane.zig 2箇所) | ~5行 |
| P5 | `std.fs.*Absolute` → `std.Io.Dir` 移行 (main/session/config) + `io` 配線 | ~80行 |
| P6 | `std.fs.File.stdout()` / `selfExePathAlloc` / `Thread.sleep` / `time.milliTimestamp` の個別対応 | ~20行 |
| P7 | `Writer.writeAll`/`print` 動作確認、posix差分対応、Writer型エラー解消 | 未知 |
| P8 | `zig build test` 通過、手動回帰、`build.zig` 側API差分（あれば） | ~10行 |
| P9 | バージョンを 0.2.0 に上げ、CHANGELOG/README 更新、コミット | — |

各Phaseは「ビルドが通る粒度」で分割し、コミットも Phase 単位で切る。

## 8. リスクと対策

| リスク | 対策 |
|---|---|
| `std.Io` API がリリースノートと実装で乖離 | 実機で `std doc` 参照しながら進める |
| stdout writer 取り方が大きく変わる | client.zig はP6で個別対応、最悪 raw posix.write fallback |
| posix syscall シグネチャ微変更 | 最終Phase P7 でエラー起点修正 |
| Writer 経由の `writeAll`/`print` が破壊される | render.zig が大量に使うため要注意、P7 で集中対応 |
| 0.16 がしばらく降りてこない | 仕様+計画だけ commit して待機。差分発生時に再評価 |

## 9. 完了条件

1. `build.zig.zon` の `minimum_zig_version = "0.16.0"`
2. `zig build` がエラー0で完了
3. `zig build test` が全テスト通過
4. `zplit new` → split → detach → attach の golden path 手動確認
5. `git tag v0.2.0` 可能な状態
